## 2. 数据入口分析：数据怎么进来的

### 2.1 Web 上传入口

**`/app/import` 页面（ImportPage）**
- 唯一的 Web 文件上传入口
- 支持拖拽和文件选择
- 调用后端 `POST /upload/knowledge` 端点
- 支持格式：`.txt`、`.md`、`.markdown`、`.pdf` 和敏感配置/证书文件
- 上传后自动触发：原件保存 → Markdown 生成 → 资产记录 → 知识文档记录 → 任务记录 → 日志写入

### 2.2 REST API 入口

| 端点 | 方法 | 功能 | 触发提取 |
|------|------|------|----------|
| `/push` | POST | 推送对话（多条消息） | 是（pipeline） |
| `/import` | POST | 导入文本/URL/文件路径 | 是（pipeline） |
| `/upload` | POST | 上传单文件（旧接口） | 是（pipeline） |
| `/upload/knowledge` | POST | 上传文件到知识库（Web-first） | 是（knowledge_base） |
| `/import/folder` | POST | 批量导入文件夹 | 是 |

### 2.3 CLI 入口

| 命令 | 功能 |
|------|------|
| `mneme add <content>` | 添加单条记忆，可选 `--extract` 触发提取 |
| `mneme import <source>` | 导入文本/URL/文件 |
| `mneme watch <path>` | 监听目录变化，自动导入 |
| `mneme migrate <source> <path>` | 从 OpenClaw/Hermes/JSON/Markdown 迁移 |

### 2.4 文件监听自动导入

- `watcher.py` 使用 `watchdog` 监听指定目录
- 支持的文件扩展名：`.txt`、`.md`、`.markdown`、`.py`、`.js`、`.ts`、`.go`、`.rs`、`.java`、`.c`、`.cpp`、`.h`、`.hpp`、`.verilog`、`.v`、`.sv`、`.yaml`、`.yml`、`.json`、`.toml`
- 文件创建/修改 → 自动 pipeline
- 文件删除 → 软删除关联记忆
- 2 秒防抖

### 2.5 Webhook / Telegram 出口（反向通知）

- `webhook.py`：新知识入库时 `POST` 到配置的 Webhook URL，HMAC-SHA256 签名
- `telegram.py`：即时通知（新知识入库）、周报摘要推送（每周一 9:00）

### 2.6 没有的入口

- **没有"从对话提取"按钮**：前端无此入口，只有后端 `/push` API 接收对话
- **没有浏览器插件剪藏**
- **没有移动端入口**

## 3. Extract 提取的用户触发方式

### 3.1 提取管道核心逻辑（`extract.py`）

提取管道 `pipeline()` 的 8 步流程：
1. SHA-256 去重
2. 存入 raw item
3. Webhook 通知
4. FTS 分块索引
5. `extract_facts()` 调用 LLM 提取事实
6. 向量化 + 向量去重
7. 存入 refined items + item_vecs
8. `extract_entities()` 提取实体 → entities 表 + entity_relations

### 3.2 用户触发方式

**自动触发（无需用户操作）**
- 文件通过 Web 上传（`/upload/knowledge`）、REST API（`/push`、`/import`、`/upload`）、CLI 导入、文件监听 → 自动进入 pipeline → 自动提取

**手动 CLI 触发**
- `mneme extract --deferred`：批量处理标记为 `pending_extraction` 的待提取项
- `mneme add "内容" --extract`：单条添加时标记提取

**智能调度触发**
- `pipeline_smart()` 根据 `ExtractionScheduler` 判断：
  - `mode=always`：始终提取
  - `mode=offpeak`：仅闲时提取（默认 22:00-08:00）
  - 非闲时只存 raw，状态标为 `pending_extraction`
  - 闲时由 `pipeline_deferred()` 批量处理

**定时维护触发**
- `cron.py` 中的 `CronScheduler` 在凌晨 3:00 执行维护，但提取本身不在 cron 内——cron 负责矛盾检测、关联发现、孤立清理、归档、回收站清理

### 3.3 没有的触发方式

- **没有 Web UI 上的"提取"按钮**
- **没有 Web UI 上的"从对话提取"按钮**
- **没有 Review 审核面板**
- 前端完全不暴露提取状态（`pending_extraction`/`active` 在 Web UI 不可见）

## 4. Review 审核工作流分析

### 4.1 当前状态

**Mneme2 没有审核（Review）工作流。**

具体事实：
- 前端 13 个页面中没有任何"待审核"列表、审核计数、批量批准/拒绝按钮
- 后端 `items` 表的 `status` 字段有：`active`、`pending_extraction`、`deleted`、`archived` —— 没有 `pending_review` 状态
- 提取出来的 refined items 直接以 `active` 状态入库，无需人工审核
- 没有审核队列、审核计数、审核提醒机制

### 4.2 最接近"审核"的交互

- **软删除 + 回收站**：用户可以通过删除来事后"否决"一条记忆
- **加工中心重试/重新解析**：用户可以对失败任务重试、对成功任务重新解析
- **回收站恢复/彻底删除**：对已删除对象做最终处置

### 4.3 没有的交互模式

- 没有待审核提醒/通知
- 没有一键批准/拒绝
- 没有批量操作（除分页外无 checkbox 多选）
- 没有审核意见/批注

## 5. Graph 图谱分析

### 5.1 后端数据结构

**entities 表**
- 字段：id, name, display_name, entity_type, properties
- entity_type 枚举：person / project / tool / concept / location / other

**entity_relations 表**
- 字段：source_id, target_id, relation, description, weight
- relation 类型：`mentioned_in`（出现在记忆中）、`co_occurrence`（在同一内容中共现）、`tag_similarity`（共享标签）、`version`（同一主题不同版本）

### 5.2 图谱构建路径

**自动构建（pipeline 内）**
- `extract_entities()` 从内容中提取实体 → 写入 entities 表
- 同时写入 `entity_relations`（`mentioned_in` 类型），source_id 是 raw item，target_id 是 entity
- entity_ids 存入 items.metadata

**定时构建（cron 维护）**
- `_discover_relations()` 基于 3 种策略补充关系：
  1. 实体共现：同一 refined item 中多个实体 → `co_occurrence`
  2. 标签相似：共享标签 ≥ 2 → `tag_similarity`
  3. 内容相似：同一标题的不同版本 → `version`

**无手动构建入口**
- 没有 Web UI 上的"构建图谱"按钮
- 没有"重新分析实体"按钮
- 没有"手动添加实体/关系"入口

### 5.3 图谱 API（后端已实现）

| 端点 | 方法 | 功能 |
|------|------|------|
| `/graph` | GET | 返回 nodes（entities + items）和 edges（entity_relations） |
| `/graph/entities` | GET | 实体列表，可按 entity_type 筛选 |
| `/graph/relations` | GET | 关系列表 |

### 5.4 图谱前端页面

**Mneme2 前端没有图谱可视化页面。**

- 路由表中没有 `/app/graph` 路由
- 没有图谱相关的 Vue 组件
- 没有 D3/Cytoscape/vis.js 等图谱可视化库的依赖
- 图谱数据只能通过后端 API 获取，无前端消费方

## 6. 新用户完整路径

### 6.1 部署启动路径

```
1. docker-compose up (或手动 mneme serve)
2. 浏览器打开 http://<host>:8866/
3. 根路径 302 → /app → 302 → /app/dashboard
4. 看到仪表盘（全部为 0）
```

**关键事实**：
- 没有注册流程（个人工具，无用户系统）
- 没有首次启动向导
- 没有 API Key 配置引导（需手动编辑 config.yaml）
- 仪表盘在无数据时显示"暂无仪表盘数据"空状态

### 6.2 从零到第一条记忆的完整流程

```
步骤1: 部署
  - 配置 config.yaml（database.path、AI 模型 key）
  - 启动服务 mneme serve 或 docker-compose up

步骤2: 访问 /app/dashboard
  - 看到空仪表盘
  - "暂无处理任务"空状态

步骤3: 点击「导入文件」或进入 /app/import
  - 拖拽文件或点击「选择文件上传」
  - 选择 .txt/.md/.pdf 文件
  - 等待上传处理

步骤4: 上传结果展示
  - 显示处理状态（success/failed）
  - 显示关联链接：资产 #id / 知识 #id / 任务 #id
  - 敏感文件标注「敏感资产：正文已遮罩」

步骤5: 回到仪表盘
  - 资产数 +1、知识文档数 +1
  - 最近任务列表出现新条目

步骤6: 进入 /app/search 搜索
  - 如果 search_index 为空，先点「重建索引」
  - 输入关键词搜索
  - 结果可跳转详情
```

### 6.3 无 LLM 配置时的路径

如果 `config.yaml` 中未配置 AI API Key：
- 文件上传仍然成功：原件保存、Markdown 生成（如为 .md 直接复制）
- `extract_facts()` 返回空列表（无 API Key 时直接返回 `[]`）
- `extract_entities()` 返回空列表
- 搜索全文索引仍可用
- 图谱为空（无实体）

### 6.4 有 LLM 配置时的路径

- 文件上传 → pipeline → LLM 提取事实 → 向量化 → 提取实体
- 如果 `extraction_schedule.mode=offpeak` 且非闲时 → raw 存为 `pending_extraction`
- 需手动 `mneme extract --deferred` 或等到闲时自动处理

## 7. 可复用的交互模式

### 7.1 列表页模式

所有列表页（资产/知识/资源/任务/版本/日志/回收站）遵循统一样板：
- `PageHeader`（eyebrow + title + description + actions 区域放筛选和操作按钮）
- 可选的 `metric-grid` 统计卡片
- 可选的筛选表单（`select`/`input` + 筛选/重置按钮）
- `DataList`（标题 + 描述 + 空状态 + actions 插槽放分页）
- `DataRow`（title/meta/detail/status/time + actions 插槽放行操作）
- 分页导航（上一页/页码/下一页）

### 7.2 危险操作模式

所有删除/恢复/重试等危险操作遵循：
1. 用户点击按钮 → `pendingAction` ref 设置为当前对象
2. `ConfirmDialog` 弹窗展示操作说明
3. 用户确认 → 调用 API → 成功/失败反馈
4. 刷新列表

### 7.3 API 调用模式

- `apiClient` 单例，统一 fetch 封装
- 超时控制（默认 15s，上传 60s）
- `ApiError` 统一错误处理
- 非 JSON 响应和 204 No Content 兼容

### 7.4 状态标签模式

`StatusPill` 组件：接受 label + variant（neutral/blue/green/orange/red）
- 通用的 `statusLabel()` 和 `statusVariant()` 函数将后端状态映射到中文标签和颜色

### 7.5 安全模式

- 所有路径只显示安全摘要（`path_summary`），不暴露宿主机绝对路径
- 敏感资产标记 `is_sensitive`，正文/路径/日志全部遮罩
- 前后端双重确认（前端 ConfirmDialog + 后端 `confirm=true`）

### 7.6 双轨 UI 模式

- `/app` 是 Vue SPA 主入口
- 每个路由有 `legacyPath` 指向旧 Jinja 页面
- 顶部栏有「Legacy 旧页面」跳转链接
- 详情页（资产/知识/资源/任务/版本的单条详情）仍由旧 Jinja 页面承担
