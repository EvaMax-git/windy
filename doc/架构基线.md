## 2. 最终产品定义

Mneme v4.1.1 的唯一正式定义如下：

```text
Mneme 是一个 Web-first、Local-first、API-first 的个人智能资产控制平面。
它统一管理资产、知识、记忆、模型/API、Agent 接入、权限、审计、备份、迁移与系统治理。
```

Mneme 不是：

1. 聊天机器人外壳。
2. 多 Agent 编排运行时。
3. 动态插件市场。
4. 第一版企业级多租户 SaaS。
5. Agent 的长期记忆主存。

Mneme 是：

1. 个人数字资产主账本。
2. 知识与记忆治理中台。
3. Agent 的受控外部记忆与上下文中心。
4. 模型/API 调用的统一网关与审计入口。
5. 可恢复、可回滚、可观测的本地优先控制平面。

## 3. 生效原则

### 3.1 五条铁律

1. PostgreSQL 是唯一事实源。
2. Redis 只做队列与派发加速，不做事实源。
3. 所有正式写入必须同事务写业务表、审计表、outbox 事件。
4. 所有外部模型/API 调用必须经 Gateway，不允许旁路直连。
5. 所有正式记忆必须经过 Candidate-first + Review，Agent 不得直接写正式 memory。

### 3.2 主归属规则

1. `Inbox` 只做待整理入口，不做长期正式主对象库。
2. `Asset` 是原件、文件元数据、入库状态的主归属。
3. `Knowledge` 是结构化正文、block、chunk、citation 的主归属。
4. `Memory` 是经 Review 生效的长期事实主归属。
5. `Review` 是横切治理中台，不属于 Memory 专属域。
6. `Gateway` 解析 capability、执行业务外发，不拥有业务对象主归属。

### 3.3 版本钉住规则

任何派生结果都必须钉住上游版本，不允许只记录对象 ID 不记录版本来源。最低要求：

1. chunk 记录 `document_version`。
2. embedding 记录 `document_version` 或 `chunk_version`。
3. graph edge 记录 `source_object_version`。
4. context pack 记录 `knowledge_version_set` 与 `memory_version_set`。
5. review item 记录 `target_version`。

## 4. 最终架构裁决

### 4.1 正式总体架构

Mneme v4.1.1 采用以下正式架构：

```text
Web UI / Agent / Personal API Client
        ↓
     /api/v4
        ↓
Core Kernel
  - Auth / Session / Agent Token
  - Policy Engine
  - Audit
  - Object Registry
  - Outbox Events
        ↓
Domain Services
  - Project / Inbox
  - Asset OS
  - Knowledge OS
  - Memory OS
  - Review Center
  - Gateway
  - Agent Center
  - Context Compiler
        ↓
PostgreSQL (source of truth)
Redis (queue accelerator)
Worker Runtime
MnemeData filesystem
External Providers via Gateway
```

### 4.2 正式技术栈

后端：

1. Python 3.12
2. FastAPI
3. Pydantic v2
4. SQLAlchemy 2
5. Alembic
6. httpx
7. Uvicorn

数据库与检索：

1. PostgreSQL 16
2. pgvector
3. PostgreSQL Full-Text Search
4. JSONB
5. GIN 索引
6. HNSW 或 IVFFlat 向量索引

异步运行时：

1. Redis
2. Dispatcher（自研，`worker/app.py`）
3. Lease Manager（自研，Redis 租约 + 心跳）
4. Retry Sweeper
5. DLQ Processor

前端：

1. Vue 3
2. TypeScript
3. Vite
4. Pinia
5. TanStack Query
6. Vue Router
7. Tailwind CSS（自定义组件，14 components）

图谱（NetworkX 后端 + 自定义 Vue 前端渲染）：

部署：

1. Docker Compose 为 v4.1.1 唯一正式基线
2. 运行单元至少包括 `web/api/worker/postgres/redis`
3. `MnemeData` 继续作为本地主文件资产根目录
4. MinIO / S3-compatible storage 仅为后续扩展，不是 v4.1.1 必选
5. Tauri / Mobile 仅做预留，不进入 M2 之前的关键路径

### 4.3 正式数据源裁决

1. 正式主库：PostgreSQL。
2. legacy 来源：Mneme2 SQLite + `MnemeData`。
3. 不允许长期双写。
4. 迁移策略固定为：并行建设 -> dry-run / shadow import -> 维护窗口切换 -> `/legacy` 只读观察期 -> legacy 退役。

## 5. 与旧文档冲突的最终裁决

### 5.1 Phase 顺序裁决

以下旧排序全部作废：

1. `Review` 在旧母方案中绑定到 Phase 5。
2. `Graph` 在旧母方案中早于 Agent 稳定闭环。
3. `Backup / Restore` 在旧母方案中被放到 Phase 8。
4. `Observability` 在旧母方案中被放到 Phase 9。

最终生效顺序以本文件第 7 节为准。

### 5.2 工期裁决

以下旧估算全部作废：

1. `MVP：10-14 周`
2. 不单列测试迁移成本
3. 不单列前端 API / 域模型重组成本

最终工期口径：

1. `M0`：6-7 周
2. `M1`：16-20 周
3. `M2`：24-30 周
4. `M3`：30-38 周

除非同时满足“5-6 FTE、需求冻结、Importer 与测试迁移无明显返工”，否则不得对外承诺 `22-26 周` 的激进工期。

### 5.3 Review 定位裁决

Review 的正式定位是“横切治理中台”，不是 Memory 附属模块。Review 至少承接：

1. Memory candidate 审批。
2. 敏感数据访问审批。
3. 高成本 provider 调用审批。
4. Importer 正式导入确认。
5. Backup Restore 确认。
6. DLQ 人工重放确认。

### 5.4 Vault 命名裁决

旧母方案中的 `api_credentials` 命名作废，统一替换为：

1. `credential_vault`
2. `vault_access_logs`

所有 provider credential、API key、OAuth token、证书、secret 都必须纳入 Vault 正式域。

### 5.5 Session 命名裁决

终审报告指出 `users / sessions` 未落正式主模型。为避免与各类 token、agent session 混淆，v4.1.1 的正式会话表命名裁决为：

1. 用户会话表：`user_sessions`
2. Agent token 表：`agent_tokens`
3. 用户个人 API token 表：后续若启用，单列 `user_api_tokens`

旧文档中的笼统 `sessions` 表述，后续一律按 `user_sessions` 实现。

### 7.3 风险闸门

下列闸门不过，禁止进入下一阶段：

1. `Audit/Outbox` 未成型：不得进入 Worker/Gateway。
2. `Backup/Restore` 未演练：不得进入正式迁移。
3. PostgreSQL 测试基线未建立：不得开始切库。
4. Review 未承接高风险动作：不得开放 Agent 正式接入与高成本外部调用。
5. Search / Context fallback 未落地：不得开放 Context Compile 正式接入。

### 7.4 Phase 1 工程门禁

终极审查中无法仅靠文档关闭的事项，统一列为 Phase 1 工程门禁。它们不阻断 Phase 1 开工，但必须在 Phase 1 验收前形成真实工程产出：

1. Alembic baseline migration 必须在 PostgreSQL 16 + pgvector 环境完成一次 `alembic upgrade head` 实跑，并提交成功日志。
2. Phase 1-3 最小 OpenAPI 或 Pydantic schema 必须落库到工程仓库，可用于生成客户端或合同测试。
3. Policy Engine 必须提供代码级接口、deny reason 枚举与权限合同测试矩阵，至少固化 `can(actor, action, object, context) -> decision` 的实现合同。

## 10. 安全与运维总裁决

### 10.1 安全

1. 主体模型固定为 `user / agent / service / system`。
2. 角色模型固定为 `owner / operator / viewer / auditor`。
3. 权限必须按 `RBAC + Project Scope + Sensitivity Ceiling + Capability Scope + Review Policy` 联合判定。
4. `secret` 级数据不得进入普通全文索引、普通 Agent 检索结果、Context Pack 原文片段、外部 provider 请求。

### 10.2 运维

运维验收不再停留在原则层，统一以本文档 §16 运维操作手册为最终量化口径，包括：

1. RPO / RTO
2. 备份恢复演练频率
3. 告警阈值
4. 最小可用性与关键链路成功率
5. 日志、指标、追踪接入要求
6. 拓扑、健康检查、发布与回滚流程

后续所有设计、接口、PR、测试、运维脚本必须满足以下规则：

1. 必须标明所属 Phase、所属对象域、是否影响本基线。

## 12. 最终结论

Mneme v4.1.1 的正式基线已经确定：

```text
以 PostgreSQL 为唯一事实源
以 Audit + Outbox + Worker + Review + Gateway + Vault 为治理骨架
以 Asset -> Knowledge -> Migration -> Memory -> Agent -> Graph 的顺序推进
以 M1 16-20 周、M2 24-30 周、M3 30-38 周作为唯一有效工期口径
```

从本文件起，Mneme 的后续开发只允许一个指挥系统，不再并存“旧母方案顺序”和“补件修正顺序”两套解释。

## 14. 规划原则与团队假设

> 来源：`Mneme_实施路线_v4.md` §2-3（已归档）

### 14.1 规划原则

1. 先搭控制平面，再扩业务平面。
2. 先让系统可回滚、可审计、可观测，再让它更聪明。
3. Graph 不是前置证明项，必须晚于 Knowledge/Memory/Agent 最小闭环。
4. 不做长期双写。
5. 不把 Importer 留到所有模块做完后才想起。

### 14.2 基线团队假设

| 角色 | 人数 |
|------|------|
| 架构/Tech Lead | 1 |
| 后端工程师 | 2 |
| 前端或全栈工程师 | 1 |
| QA/DevOps 共享支持 | 0.5 |

在此基线下现实工期：
- **M1**（资产/知识切换可上线）：16-20 周
- **M2**（Memory + Agent 最小闭环）：24-30 周
- **M3**（Graph/Eval/高级观测/移动预留）：30-38 周

若只有单主程或需求持续变动，整体周期应按 40+ 周看待。

### 14.3 风险闸门

以下闸门不过，禁止进入下一阶段：

| 闸门 | 不通过后果 |
|------|-----------|
| Audit/Outbox 未成型 | 不能进入 Worker/Gateway |
| Backup/Restore 未演练 | 不能进入正式迁移 |
| PG 测试基线未建立 | 不能开始切库 |
| Review 未承接高风险动作 | 不能开放 Agent 和外部高成本调用 |
| Search/Context fallback 未落地 | 不能开放 Agent 正式接入 |

## 15. Phase 0-8+ 详细路线

> 来源：`Mneme_实施路线_v4.md` §4-6（已归档）。Phase 顺序与里程碑见本文 §7。

### Phase 0 — 冻结边界 + 文档定稿（1-1.5 周）

**目标**: 彻底定死 Mneme2 与 v4.1 边界，停止继续扩张旧 `items/chunks` 主线，输出可执行的迁移/一致性/安全/路线/成本文档。

### Phase 1 — Core Platform（2-3 周）

**目标**: 建立 v4.1 正式技术底座。先落审计、身份、事件 outbox、最小观测。

关键交付:

### Phase 3 — Storage + Project + Inbox + Asset OS + Pipeline（3-4 周）

**目标**: 把原件和任务主线迁到新平台，建立新 Asset OS 和 Pipeline 运行边界。

### Phase 4 — Knowledge OS + Retrieval + Citation（4-5 周）

**目标**: 形成资产到知识的正式结构化主线，做实"编辑后的一致性"。

验收: 编辑 block 后正文立即正确；搜索/embedding即使未追平也有明确 stale 状态；Citation 定位到 block/source map 级别。

### Phase 5 — Importer + 测试迁移 + 资产/知识切换（3-4 周）

**目标**: 把 Mneme2 真实生产资产迁入新平台，完成 SQLite→PG 第一阶段正式切换。

## 17. 未来能力预设计 — 多模态 Embedding 管道

> 来源：`doc/engineering/L7-multimodal-embedding.md`（已吸收）
>
> **状态: 预设计（尚未实现）** | 版本: v0.1 | 日期: 2026-05-08

### 17.1 设计目标

在现有文本 embedding 管道（`mneme/memory/embedding.py`，已实现）基础上，扩展为**多模态 embedding 管道**：

| 模态 | 输入类型 | 用途 | 优先级 |
|------|---------|------|--------|
| 文本 | `str` | 记忆、知识文档、消息的语义向量 | ✅ 已实现 |
| 图片 | `bytes` / path / URL | 资产缩略图、知识文档插图的语义向量 | 🔲 Phase 2 |
| 音频 | `bytes` / path | 语音记录、会议录音的语义向量 | 🔲 预留 |
| 视频 | `bytes` / path | 演示录制、监控片段 | 🔲 远期预留 |
| 混合 | `(text, image)` | 图文联合检索、多模态 RAG | 🔲 远期预留 |

**设计原则**: 统一接口（`embed()` 入口）/ Provider 抽象（经 Gateway）/ 维度中立 / 降级策略（多模态不可用时透明降级到文本）/ 索引统一（同张 `memory_index_entries` 表，`modality` 列区分）。

### 17.2 数据模型扩展（未实现）

`memory_index_entries` 表新增: `modality`(varchar, 默认'text')、`source_asset_id`(uuid)、`mime_type`、`file_hash`、`file_size_bytes`。

新表 `embedding_models`: model_id/model_code/provider/capabilities/input_modalities/vector_dimensions/max_input_length/config_json/status。

新表 `multimodal_assets`: asset_id/project_id/modality/original_filename/mime_type/file_hash/file_size_bytes/storage_path/thumbnail_path/width/height/duration_seconds/text_extracted/metadata_json。

### 17.3 管道架构

```
统一入口: embed() → Modality Router
  ├── Text Pipeline (现有)
  ├── Image Pipeline (新建): 格式检测(PIL) → 缩略图 → OCR(可选) → 哈希去重 → Gateway Embedding
  ├── Audio Pipeline (预留): 格式检测(ffmpeg) → ASR(可选) → Embedding
  └── Multimodal Fusion (远期): 分别编码 → 投影到同一空间 → 归一化
       ↓
  Gateway Call (embedding.create)
       ↓
  Index Entry Persistence (memory_index_entries)
       ↓
  Search Integration (按modality过滤 + 向量相似度)
```

### 17.4 图片管道（Image Pipeline）

输入 → PIL格式检测（JPEG/PNG/WebP/GIF首帧/BMP/TIFF，转RGB，最大1024×1024）→ 缩略图（256×256）→ OCR文字提取（Tesseract/PaddleOCR，可选）→ SHA-256哈希去重 → Gateway Embedding（CLIP/SigLIP模型）。

### 17.7 风险与依赖

| 风险 | 等级 | 缓解 |
|------|------|------|
| Gateway 不支持多模态模型 | 高 | 先做文本降级，图片 embedding 走独立 provider |
| 图片 embedding 质量不足 | 中 | 支持多模型切换 |
| OCR 依赖外部库 | 中 | 封装为可选能力 |
| 向量维度不一致 | 中 | 投影层统一维度 |
| 搜索延迟增加 | 低 | 图片 embedding 缓存 + 异步预处理 |
