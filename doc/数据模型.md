## 2. 建模总原则

### 2.1 命名原则

1. 统一使用 `snake_case`。
2. 主键统一使用 `uuid`。
3. 时间字段统一使用 `timestamptz`。
4. JSON 扩展字段统一使用 `jsonb`。
5. 业务状态统一使用受控 `varchar` + `CHECK`，不依赖匿名自由文本。

### 2.2 一致性原则

1. 正式写入必须同事务写业务表、`audit_events`、`events`。
2. `event_deliveries` 记录事件投递结果，解决幂等消费与重复派发审计。
3. `dead_letters` 是 PostgreSQL 中的正式 DLQ 事实源。
4. `budget_tracking` 记录预算预留、确认、释放、结算。
5. `usage_limits` 记录预算与速率限制规则，不把额度散落在各业务表。

### 2.3 安全原则

1. `user_sessions` 与 `agent_tokens` 物理分离。
2. `credential_vault` 只存密文与包装密钥，不存明文。
3. `vault_access_logs` 必须覆盖允许、拒绝、导出、轮换、调用等行为。
4. `event_source` 与 `messages` 独立建模，避免把聊天记录与提炼结果混写。

## 4. 核心字段设计说明

### 4.1 `users`

用途：

1. 人类用户主体。
2. Owner bootstrap 后的正式身份对象。
3. Web Session、个人 API Token、审计 actor 的来源。

关键规则：

1. `status` 仅允许 `active / disabled / locked / pending_bootstrap`。
2. `role_code` 仅允许 `owner / operator / viewer / auditor`。
3. `password_hash` 不可为空。
4. `mfa_mode` 支持 `none / totp / passkey / required_but_unbound`。

### 4.2 `user_sessions`

用途：

1. Web 会话正式表。
2. 支持服务端撤销、step-up、设备追踪、最近活跃时间。

关键规则：

1. 不与 Agent Token 共表。
2. `step_up_verified_at` 为空时，视为未完成高风险动作二次认证。
3. `expires_at` 必须显式设置。

### 4.3 `event_deliveries`

用途：

1. 记录每个 `event_id + consumer_name` 的派发过程。
2. 支持幂等消费、重试、失败诊断。

关键规则：

1. `UNIQUE(event_id, consumer_name)` 必须存在。
2. `delivery_state` 仅允许 `pending / dispatched / acknowledged / failed / dead_letter`。

### 4.4 `dead_letters`

用途：

1. PostgreSQL 内的正式 DLQ。
2. 对接 Review、人工重放、审计、告警。

关键规则：

1. 不只记录错误文本，还要记录 `failure_class`、`retry_exhausted`、`external_effect_state`。
2. `replay_state` 仅允许 `pending / under_review / replayed / cancelled / resolved`。

### 4.5 `credential_vault`

用途：

1. 外部 provider 凭据唯一正式存放处。
2. 支持 envelope encryption 与轮换。

关键规则：

1. 不存明文 secret。
2. `ciphertext`、`key_wrap`、`key_version` 必填。
3. `scope_json` 必须表达 capability、project、sensitivity 范围。

### 4.6 `vault_access_logs`

用途：

1. 记录 Vault 访问与使用。
2. 记录允许、拒绝、轮换、导出、取用、禁用等审计动作。

关键规则：

1. 无论允许还是拒绝都必须写日志。
2. 不记录明文 secret，仅记录摘要与结果。

### 4.7 `conversations / event_source / messages`

用途：

1. `conversations` 表示一个正式会话容器。
2. `event_source` 记录来源平台、会话外部 ID、参与者、时间窗口、抓取批次。
3. `messages` 记录单条原始消息或事件片段。

关键规则：

1. 原始消息和 Candidate/Memory 必须分离。
2. `messages` 至少记录 `role_code`、`content_text`、`sensitivity_level`、`message_time`。
3. `event_source` 允许一个 conversation 对接多个来源段。

### 4.8 `usage_limits / budget_tracking`

用途：

1. `usage_limits` 记录预算、速率、阈值、审批门槛。
2. `budget_tracking` 记录预留、确认、释放、拒绝、退款等预算流水。

关键规则：

1. 所有外部 provider 调用必须先有预算评估。
2. 高成本调用的预算拒绝必须可审计、可解释。

## 5. PostgreSQL DDL 完整基线

本节是 Alembic baseline 的权威输入。DDL 必须按以下顺序执行，不允许把外键、触发器或依赖后置对象混入建表层：

1. 基础层：扩展、函数、无外键表。
2. 核心层：Core/Auth/Audit/Event/Review/Gateway/Jobs/Pipeline/Asset/Knowledge/Memory/Context/Object 全部 `CREATE TABLE`，只保留主键、唯一约束、CHECK 约束与默认值。
3. 约束层：所有 `FOREIGN KEY` 统一用 `ALTER TABLE ... ADD CONSTRAINT` 后置添加，解决前向引用与循环引用。
4. 索引层：普通索引、GIN、vector 索引。
5. 触发器层：所有表存在后再创建 `updated_at` 触发器。

可执行性声明：本文 DDL 已按分层结构编排为“基础 -> 核心/领域建表 -> 约束 -> 索引 -> 触发器”，所有前向 `FOREIGN KEY` 均已后置为 `ALTER TABLE ... ADD CONSTRAINT`，可直接作为 Alembic baseline migration 的参考输入。已知验证项：仍需在真实 PostgreSQL 16 + pgvector 环境执行空库升级，确认扩展安装、vector 类型、GIN/vector 索引、触发器与全部约束一次通过。

### 5.0 基础层

```sql
BEGIN;

CREATE EXTENSION IF NOT EXISTS pgcrypto;
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS vector;

CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS trigger
LANGUAGE plpgsql
AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$;
```

### 5.5 Raw Event 裁决

`raw_events` 是 Phase 6 起正式原始事件事实表，不替代 `event_source/messages`：

1. `event_source` 描述来源批次、外部会话、参与者、时间窗口和导入上下文。
2. `messages` 保存可直接作为聊天记录呈现的单条消息正文。
3. `raw_events` 保存原始平台事件或导入记录，覆盖 tool call、reaction、attachment、system event 等非消息事件，也可用 `message_id` 指向由该原始事件规范化出的消息。
4. `memory_candidates.source_type='raw_event'` 时，`source_id` 必须指向 `raw_events.raw_event_id`。
5. `memory_sources.raw_event_id` 是正式外键，用于把正式 memory 版本追溯到原始事件；证据范围使用 `source_span` 记录文本 offset、JSON path、附件片段或消息 span。
6. `raw_events.payload_json` 保存脱敏或受控原文；`payload_hash + idempotency_key` 负责去重与幂等。`secret` 级 payload 默认不得进入普通 FTS/vector、Context Pack 原文片段或外部 provider 请求。

### 5.6 Memory 检索索引裁决

`memory_index_entries` 是 Memory Search 与 Context Compiler 的正式检索索引表：

1. 索引粒度固定为 `memory_id + memory_version + index_profile`。
2. `content_hash` 由 `memory_text + sensitivity_level + index_profile + embedding_model_id` 派生，用于避免重复索引。
3. `fts_state/vector_state` 使用 `pending/ready/stale/failed`，与 `index_states` 词表一致。
4. `embedding_model_id` 指向 `provider_models`；v4.1.1 默认只支持 `vector(1536)` 全局 profile，多模型维度并存进入后续专项 migration。
5. Search 降级规则：vector 未 ready 时允许只用 FTS；FTS 和 vector 均未 ready 时返回 `meta.index_state` 并仅允许按 canonical_key/title 精确过滤。

### 5.7 Context Pack 审计裁决

Context Compiler 必须落 `context_packs/context_pack_items`，不得只依赖 `api_call_logs` 摘要：

1. `context_packs` 记录请求、Agent、项目、编译模式、预算、版本集合和保留期。
2. `context_pack_items` 记录纳入或排除的知识/记忆/raw_event/fallback_query 项、版本、顺序、理由、token 估算、内容摘要。
3. `api_call_logs` 只记录外部调用摘要；如 Context Pack 被用于 Gateway 调用，可通过 `context_packs.api_call_log_id` 关联。
4. 默认保留期与 Gateway 调用日志一致为 180 天；清理后必须保留 `context_pack_id`、版本集合摘要、排除原因统计和审计事件。

## 6. 关系与实现说明

### 6.1 认证与主体关系

1. `users 1:n user_sessions`
2. `users 1:n agent_tokens` 通过 `issued_by_user_id`
3. `users 1:n agents` 通过 `owner_user_id`
4. `agent_tokens` 不复用 `user_sessions`

### 6.2 事件与投递关系

1. `events 1:n event_deliveries`
2. `event_deliveries -> dead_letters` 通过 `source_type=event_delivery`
3. 所有正式写 API 必须先写业务对象，再写 `audit_events` 与 `events`

### 6.3 Vault 与 Gateway 关系

1. `providers 1:n credential_vault`
2. `capabilities` 决定调用能力
3. `vault_access_logs` 记录调用/拒绝/轮换
4. 外部 provider 调用必须与 `usage_limits`、`budget_tracking` 联动

### 6.4 聊天与记忆关系

1. `conversations 1:n event_source`
2. `conversations 1:n messages`
3. `messages` 可生成 `memory_candidates`
4. `memory_candidates` 经 Review 才可进入 `memories`

### 6.5 知识与上下文关系

1. `knowledge_documents 1:n knowledge_blocks`
2. `knowledge_documents 1:n knowledge_chunks`
3. `index_states` 记录索引就绪与滞后状态
4. Context Compile 必须返回所引用的知识文档/记忆版本集合

## 8.5 Memory 操作契约

Memory 写入只允许通过 Review 驱动的操作 API，不允许 Agent 或抽取 Pipeline 直接写 active memory。所有操作必须同事务写 `memories`、`memory_versions`、必要的 `memory_sources/memory_relations/memory_index_entries` 状态、`audit_events` 与 `events`。

| 操作 | API | 入口状态 | 主要写入 | 版本规则 |
| --- | --- | --- | --- | --- |
| approve/create | `POST /api/v4/memory/candidates/{candidate_id}/approve-create` | `memory_candidates.pending_review` + `review_items.approved` | 新建 `memories(status='active', activated_by_review_item_id)`、`memory_versions(action='create', version=1)`、`memory_sources` | 从 1 开始 |
| approve/update | `POST /api/v4/memory/candidates/{candidate_id}/approve-update` | 目标 memory active/expired 且 review approved | 更新 `memories.memory_text/current_version`、插入 `memory_versions(action='update')` | `current_version + 1` |
| mark_conflict | `POST /api/v4/memories/{memory_id}/mark-conflict` | active/expired | 插入 `memory_relations(relation_type='conflicts_with')`，候选可置 `conflict` | 不改正文时不递增 memory version |
| merge | `POST /api/v4/memories/{memory_id}/merge` | 两个 active memory + review approved | 目标 memory 新版本、来源 memory `status='merged'`、`memory_relations(merged_into)` | 两侧均写版本 |
| expire | `POST /api/v4/memories/{memory_id}/expire` | active | `status='expired'`、`expired_at`、`memory_versions(action='expire')` | 递增版本 |
| restore | `POST /api/v4/memories/{memory_id}/restore` | expired/deleted | 恢复 active 必须带 `review_item_id`，写 `memory_versions(action='restore')` | 递增版本 |
| delete | `DELETE /api/v4/memories/{memory_id}` | active/expired/merged | `status='deleted'`、`memory_versions(action='delete')` | 递增版本 |

约束：

1. active memory 必须满足 `activated_by_review_item_id IS NOT NULL`。
2. `approve/create` 与 `approve/update` 的幂等键由 `review_item_id + candidate_id + target_memory_id? + action` 派生。
3. merge 不物理删除来源 memory；Context/Search 默认过滤 `status in ('merged','deleted')`。
4. expire 后默认不进入 Context Pack，除非请求显式 `include_expired=true` 且权限允许。
5. restore 必须重新触发 `memory_index_entries`，旧索引置 `stale`，新版本置 `pending`。

规则：

1. `candidate_hash` 由 `project_id + normalized_candidate_text + sensitivity_level + evidence_hash` 生成，并落 `memory_candidates(candidate_hash)`。
2. evidence spans 必须同步落 `memory_sources.source_span` 或候选 `metadata_json.evidence_spans`，审批转正时复制到正式 `memory_sources`。
3. Pipeline 重跑使用同一 `idempotency_key` 时不得重复创建候选；内容不一致返回 `409 idempotency_conflict` 并写 `pipeline_runs.error_json`。
4. Dedup 命中既有 active memory 时，只能生成 update/merge 建议，不得自动改写正式 memory。
5. 每次 LLM 抽取必须写 `api_call_logs`、`budget_tracking`、`audit_events(action='memory.extract')`。

## 9. Phase 2-3 关系补充说明

### 3.1 Job / Event / DLQ

1. `jobs.job_key` 是 Worker 重入与幂等执行的唯一键，对应一致性设计中的 `job_type + aggregate_id + target_version + cause_event_id`。
2. `jobs.cause_event_id -> events.event_id` 表示 Job 由 outbox 事件驱动。
3. 执行失败且重试耗尽时，事实进入既有 `dead_letters`，使用 `dead_letters.source_type='job'` 与 `dead_letters.source_id=jobs.job_id` 关联。
4. `job_logs` 只记录 Job 内部步骤日志，不替代 `audit_events`。

### 3.2 Review / Audit

1. `review_items` 是横切治理中台表，可审核 Memory candidate、敏感访问、高成本调用、导入确认、恢复确认、DLQ replay。
2. `audit_events.review_item_id` 在补件中补上外键，用于把审批导致的业务动作与审批项关联。
3. `review_items.target_type/target_id/target_version` 保留多态引用，满足统一基线的版本钉住要求。

### 3.3 Inbox / Asset / Knowledge

1. `inbox_items` 是待整理入口，不是长期正式主对象库。
2. `assets` 是原件、文件元数据、入库状态的正式主账本。
3. `asset_metadata` 承载可扩展元数据；高频关键状态仍保留在 `assets` 上。
4. `source_maps` 用于把 Asset、Knowledge、Message、Memory 之间的来源关系钉住到 span 级别。

### 3.4 Pipeline / Job

1. `pipeline_defs` 记录可执行 Pipeline 定义与版本。
2. `pipeline_runs` 记录一次 Pipeline 执行的总体状态。
3. `pipeline_runs.root_job_id -> jobs.job_id` 表示 Pipeline Run 的根执行 Job；细粒度步骤可先落在 `job_logs`，后续如需要再扩展 `pipeline_step_runs`。

### 3.5 Memory Version / Source

1. `memory_versions` 是正式 Memory 的领域版本流，不替代全局 `object_versions` 预留。
2. `memory_sources` 把正式 Memory 版本关联到 candidate、message、asset、document、block 与 raw event。
3. `raw_events` 已作为 Phase 6 正式原始事件表落入基线，`memory_sources.raw_event_id` 必须通过外键指向 `raw_events.raw_event_id`。

## 10. 状态机与关系边界

### 10.1 `api_call_logs`

状态机：

```text
planned
→ budget_reserved
→ credential_checked
→ in_flight
→ succeeded

planned / budget_reserved / credential_checked
→ denied

in_flight
→ failed / timeout / cancelled / dead_letter
```

规则：

1. `idempotency_key` 是 Gateway 调用级幂等键，必须由 `capability_id + project_id + actor + request_fingerprint` 派生或由调用方显式传入。
2. `provider_request_fingerprint` 是脱敏请求指纹，不包含原始 prompt、明文 secret 或完整文件内容。
3. `request_summary` 与 `response_summary` 只允许存摘要、token、模型参数、截断片段和错误摘要；敏感原文应留在业务对象或加密存储中。
4. 保留期为 180 天，`retention_until` 默认 `created_at + 180 days`；清理任务必须保留聚合指标，不保留原始调用摘要。

与其他表关系：

1. `budget_tracking` 记录预算预留、确认、释放、拒绝；`api_call_logs.budget_tracking_id` 指向本次调用的预算事实。
2. `audit_events` 记录谁以什么权限上下文发起、拒绝或完成调用；`api_call_logs` 不替代审计。
3. `events` 记录可投递领域事件，例如 `gateway.call.succeeded`、`gateway.call.failed`；`api_call_logs.event_id` 指向对应 outbox 事件。
4. `vault_access_logs` 记录凭据取用或拒绝；`api_call_logs.vault_access_log_id` 指向本次凭据访问日志。
5. 重试耗尽或外部副作用未知时，`dead_letters.source_type='provider_call'` 且 `source_id=api_call_logs.api_call_log_id`。

### 10.2 `provider_models / capability_bindings`

`provider_models.status` 状态机：

```text
active
→ degraded
→ active

active / degraded
→ deprecated
→ disabled

active / degraded
→ disabled
```

状态含义：

1. `active`：可被 binding 正式选择。
2. `degraded`：仅可作为降级候选，调用方响应必须标记降级。
3. `deprecated`：保留历史调用查询，不再创建新 binding。
4. `disabled`：禁止新调用，历史 `api_call_logs` 仍保留引用。

`capability_bindings.status` 状态机：

```text
active
→ degraded
→ active

active / degraded
→ shadow
→ active

active / degraded / shadow
→ disabled
```

状态含义：

1. `active`：可承接正式流量。
2. `degraded`：仅在主 binding 不可用时被 Gateway 选中。
3. `shadow`：只记录候选决策或压测，不承接正式外发。
4. `disabled`：禁止被 Gateway 选中。

关系：

1. `providers` 表示外部服务商或本地服务端点。
2. `provider_models` 表示 provider 下可调用的具体模型或能力端点。
3. `capabilities` 表示 Mneme 内部能力抽象，例如 chat、embedding、ocr、search、export。
4. `capability_bindings` 把 capability 绑定到 provider、provider_model、credential 与 project/sensitivity scope。
5. `credential_vault.scope_json` 是凭据本身允许的范围；`capability_bindings.policy_json` 是 Mneme 调度和治理策略。两者必须同时满足。

绑定选择顺序：

1. 优先匹配 `project_id + sensitivity` 的 `active` binding。
2. 再匹配项目级 binding。
3. 再匹配全局 binding。
4. 同一层按 `priority ASC` 选择；状态为 `degraded` 时仅可作为降级候选。

敏感度规则：

1. 请求对象的 `sensitivity_level` 必须落在 `sensitivity_floor` 与 `sensitivity_ceiling` 之间。
2. `provider_models.sensitivity_ceiling`、`credential_vault.scope_json.sensitivity_ceiling`、`capability_bindings.sensitivity_ceiling` 三者取最严格值。
3. 超过阈值时返回 `403 permission_denied` 或 `422 review_required`，不得降级为旁路直连。

### 10.3 `object_registry / object_versions`

`object_registry.status` 状态机：

```text
active
→ archived
→ active

active / archived
→ superseded

active / archived / superseded
→ deleted

active
→ quarantined
→ active / archived / deleted
```

状态含义：

1. `active`：对象可被正常读取、索引、引用。
2. `archived`：对象不可进入新索引任务，但历史引用仍有效。
3. `superseded`：对象被新对象取代，必须通过 `source_maps` 或领域关系记录替代关系。
4. `quarantined`：对象存在安全、合规或解析风险，禁止进入 Gateway、Pipeline 和 Context。
5. `deleted`：逻辑删除；物理清理由保留策略执行。

`object_versions.action` 动作流：

```text
create
→ update / merge / expire / archive / supersede / delete
archive
→ restore / delete
delete
→ restore
```

动作规则：

1. 每个对象必须从 `create` 的 version 1 开始。
2. `update/merge/expire/archive/delete/restore/supersede/import` 都必须递增版本。
3. `import` 只用于迁移或外部导入落账；导入后对象仍应有正式 `create` 或领域等价动作。

边界：

1. `object_registry` 是全局对象登记簿，提供统一对象 ID、类型、项目、状态、敏感度、当前版本；`object_registry.object_id` 裁决为等于领域对象主键，不另设映射 ID。
2. `object_versions` 是跨域版本索引和审计快照，不替代领域版本表。
3. `memory_versions` 是 Memory 域的正式版本流；`object_versions` 只保存跨域通用快照、事件和审计指针。
4. 未来文档、资产、图谱若有专属版本表，应以领域表为事实源，以 `object_versions` 作为统一查询与回滚索引。
5. `events.aggregate_type/aggregate_id/aggregate_version` 必须与 `object_registry.object_type/object_id/current_version` 对齐。
6. `source_maps` 负责来源到目标的 span 级映射；`object_versions.source_map_id` 只记录本版本由哪条来源映射直接触发，不承载完整引用图。

事务要求：

1. 创建正式主对象时，同事务写领域表、`object_registry(current_version=1)`、`object_versions(action='create')`、`audit_events`、`events`。
2. 更新正式主对象时，同事务递增领域表版本、更新 `object_registry.current_version`、插入 `object_versions`、写 `audit_events` 与 `events`。
3. 归档、删除、恢复均必须写 `object_versions`，不得只改领域表状态。

| POST | `/api/v4/vault/credentials/{credential_id}/rotate` | `new_secret`、`reason` | `credential_id`、`status`、`rotated_at`、`access_log_id` | `200`、`400`、`401`、`403`、`404` |
| POST | `/api/v4/vault/credentials/{credential_id}/disable` | `reason` | `credential_id`、`status='disabled'`、`access_log_id` | `200`、`401`、`403`、`404` |
| GET | `/api/v4/vault/access-logs` | query: `credential_id?`、`provider_id?`、`result?`、`after?`、`limit?` | `logs[]` | `200`、`401`、`403` |

安全约束：

1. `secret` 只允许入站，不允许在响应中返回。
2. `credential_material_ref` 只可由 Gateway 内部使用，外部 API 客户端不得获得明文。
3. 所有 create/use/rotate/disable/export/denied 行为必须写 `vault_access_logs` 与 `audit_events`。

### 11.5 Backup / Restore Confirm API

| 方法 | 路径 | 请求字段 | 响应字段 | 状态码 |
| --- | --- | --- | --- | --- |
| POST | `/api/v4/backups` | `backup_scope`、`include_vault_ciphertext`、`target`、`reason`、`idempotency_key` | `job_id`、`status`、`review_item_id?` | `202`、`400`、`401`、`403`、`409`、`422` |

确认规则：

1. Restore 必须 step-up；高风险 restore 必须 Review。

3. Vault 明文不得进入普通 backup；Vault 密文导出必须记录访问日志。

## 12. Phase 3 核心 API 契约

### 12.1 Inbox / Asset 入库 API

| 方法 | 路径 | 请求字段 | 响应字段 | 状态码 |
| --- | --- | --- | --- | --- |
| POST | `/api/v4/inbox/items` | `project_id?`、`inbox_type`、`source`、`source_uri?`、`source_ref?`、`title?`、`payload_json`、`metadata_json?`、`idempotency_key` | `inbox_item_id`、`status='received'`、`content_hash?` | `201`、`400`、`401`、`403`、`409` |
| POST | `/api/v4/inbox/items/{inbox_item_id}/stage` | `metadata_json?`、`expected_status?` | `inbox_item_id`、`status='staged'` | `200`、`401`、`403`、`409`、`404` |
| POST | `/api/v4/inbox/items/{inbox_item_id}/confirm` | `asset_type`、`title`、`storage_ref`、`content_hash`、`sensitivity_level`、`retention_policy?`、`metadata_json?`、`start_pipeline?` | `asset_id`、`asset_uid`、`status`、`ingest_state`、`pipeline_run_id?` | `201`、`400`、`401`、`403`、`409`、`422` |
| GET | `/api/v4/assets/{asset_id}` | none | `asset`、`metadata[]`、`source_maps[]?` | `200`、`401`、`403`、`404` |

入库事务规则：

1. Confirm 必须同事务写 `assets`、回填 `inbox_items.asset_id/status`、写 `object_registry/object_versions`、`audit_events`、`events`。
2. 若 `start_pipeline=true`，同事务创建 `pipeline_runs` 或 `jobs` 的可追踪入口。
3. 内容 hash 冲突时返回 `409 duplicate_asset`，响应 `details.existing_asset_id`。

### 12.2 Asset 归档 / 去重 / 失败重试 API

| 方法 | 路径 | 请求字段 | 响应字段 | 状态码 |
| --- | --- | --- | --- | --- |
| POST | `/api/v4/assets/{asset_id}/archive` | `reason`、`expected_version` | `asset_id`、`status='archived'`、`current_version`、`event_id` | `200`、`401`、`403`、`409`、`404` |
| POST | `/api/v4/assets/deduplicate` | `project_id`、`content_hash`、`strategy`、`candidate_asset_ids?` | `duplicate_of_asset_id?`、`affected_asset_ids[]`、`actions[]` | `200`、`400`、`401`、`403`、`409` |
| POST | `/api/v4/assets/{asset_id}/retry-ingest` | `reason`、`pipeline_def_id?`、`idempotency_key` | `asset_id`、`ingest_state`、`job_id?`、`pipeline_run_id?` | `202`、`401`、`403`、`409`、`404` |
| POST | `/api/v4/inbox/items/{inbox_item_id}/retry` | `reason`、`idempotency_key` | `inbox_item_id`、`status`、`job_id?` | `202`、`401`、`403`、`409`、`404` |

状态规则：

1. 归档只允许 `active -> archived`，恢复需走单独 restore/import 策略。
2. 去重不得删除原件；默认策略是归档重复 asset 并写 `source_maps(mapping_role='derived_from')`。
3. Retry 只能从 `failed` 或可重入的 `pending/staged` 状态进入。

### 12.3 Pipeline 定义 / 运行 / 取消 / 重跑 / 日志 API

| 方法 | 路径 | 请求字段 | 响应字段 | 状态码 |
| --- | --- | --- | --- | --- |
| POST | `/api/v4/pipeline-defs` | `project_id?`、`pipeline_code`、`pipeline_type`、`version?`、`name`、`description?`、`config_json` | `pipeline_def_id`、`status='draft'`、`version` | `201`、`400`、`401`、`403`、`409` |
| POST | `/api/v4/pipeline-defs/{pipeline_def_id}/publish` | `expected_version` | `pipeline_def_id`、`status='active'`、`event_id` | `200`、`401`、`403`、`409`、`404` |
| POST | `/api/v4/pipeline-runs` | `pipeline_def_id`、`project_id?`、`trigger_type`、`target_type?`、`target_id?`、`target_version?`、`input_json`、`idempotency_key` | `pipeline_run_id`、`status='pending'`、`root_job_id?` | `202`、`400`、`401`、`403`、`409` |
| POST | `/api/v4/pipeline-runs/{pipeline_run_id}/cancel` | `reason`、`expected_status?` | `pipeline_run_id`、`status='cancelled'`、`event_id` | `200`、`401`、`403`、`409`、`404` |
| POST | `/api/v4/pipeline-runs/{pipeline_run_id}/rerun` | `mode`、`from_failed_step?`、`idempotency_key` | `pipeline_run_id`、`new_pipeline_run_id`、`root_job_id?` | `202`、`400`、`401`、`403`、`409`、`404` |
| GET | `/api/v4/pipeline-runs/{pipeline_run_id}` | none | `pipeline_run`、`root_job?`、`summary` | `200`、`401`、`403`、`404` |
| GET | `/api/v4/pipeline-runs/{pipeline_run_id}/logs` | query: `level?`、`after?`、`limit?` | `pipeline_run_id`、`logs[]` 来自 `job_logs` | `200`、`401`、`403`、`404` |

运行规则：

1. Pipeline run 的总体事实在 `pipeline_runs`；Phase 3 不强制 `pipeline_step_runs`。
2. 步骤日志在 `job_logs`；UI 若需要步骤级状态，只能展示 root job 与日志汇总。
3. Cancel 必须传播到 root job；无法取消外部副作用时写 `dead_letters` 或 Review。
4. Rerun 必须创建新的 `pipeline_runs`，不得复写旧 run 的最终状态。

## 13. 验收口径

本文档补齐后，终审 v3 的 6 个缺口对应如下：

1. `api_call_logs`：已给出完整 DDL、状态机、幂等键、索引、180 天保留期、与预算/审计/outbox/vault 的关系。
2. `provider_models / capability_bindings`：已给出完整 DDL，并定义 provider、model、capability、credential、project/sensitivity scope 关系。
3. `object_registry / object_versions`：已给出 Phase 1 必需 DDL，并说明与 `events`、领域版本表、`source_maps` 的边界。
4. Phase 2 API：已覆盖 Job runtime、Review v1、Gateway call、Vault access、Backup/Restore confirm。
5. Phase 3 API：已覆盖 Inbox/Asset 入库、Asset 归档/去重/失败重试、Pipeline 定义/运行/取消/重跑/日志查询。
6. 后续 Phase 表裁决：见 `架构基线.md` §14-15（Phase 路线 + 实施记录）。
