## 1. 目标

本文件定义 Mneme v4.1.1 的控制平面约束，解决以下 P0 缺口：

1. 哪些写入在同一事务里完成，哪些只能异步完成。
2. 事件如何可靠投递，如何避免重复消费。
3. Worker、Queue、Dead Letter 如何设计。
4. 模块失败时如何回滚、补偿和降级。
5. 跨 `Asset / Knowledge / Memory / Review / Gateway / Context / Graph` 如何保证一致性。

## 2. 一致性原则

## 2.1 一致性分层

Mneme 不是分布式事务系统，不能追求“所有模块全强一致”。正式标准分三层：

### A. 聚合内强一致

以下内容必须在一个 PostgreSQL 事务内完成：

1. 业务主对象变更。
2. 对应版本号推进。
3. 审计事件落库。
4. outbox 事件写入。
5. 同事务内必须存在的 review / policy 决策结果。

### B. 派生读模型最终一致

以下对象允许异步追平：

1. `knowledge_chunks`
2. `embeddings`
3. `index_states`
4. `graph_nodes/graph_edges`
5. `context_packs`
6. `notifications`

要求不是“立即一致”，而是：

1. 状态可见。
2. 版本可追踪。
3. 失败可重试。
4. 旧稳定结果在新结果未完成前仍可读。

### C. 外部能力调用非事务一致

外部 provider、OCR、embedding、LLM、webhook 不参与数据库事务。对它们采用：

```text
本地事务提交
→ outbox 派发
→ worker 执行外部调用
→ 结果回写
→ 失败补偿/降级
```

不允许“持有数据库事务再等待外部 API”。

## 2.2 核心结论

1. PostgreSQL 是唯一事实源。
2. Redis 只做队列加速器，不做事实源。
3. `events` 表承担 outbox 职责。
4. `jobs` 表承担执行状态职责。
5. 审计事件和业务变更同事务提交。
6. Graph、Embedding、Context 失败不能阻塞 Asset/Knowledge 主链路。

## 3. 统一事件模型

## 3.1 事件表最小字段

`events` 至少包含：

| 字段 | 说明 |
| --- | --- |
| `event_id` | 全局唯一 UUID |
| `event_type` | 例如 `asset.created`、`block.updated` |
| `aggregate_type` | 业务聚合类型 |
| `aggregate_id` | 聚合主键 |
| `aggregate_version` | 该事件对应的对象版本 |
| `correlation_id` | 一次业务链路的总关联 ID |
| `causation_id` | 触发它的上游事件或请求 ID |
| `idempotency_key` | 去重键 |
| `producer` | 产生事件的模块 |
| `payload_json` | 业务载荷 |
| `visibility` | internal / external / audit_only |
| `publish_state` | pending / dispatched / delivered / failed / dead_letter |
| `occurred_at` | 业务发生时间 |
| `committed_at` | 事务提交时间 |

## 3.2 事件分类

### 事务内事件

只写库、不出队列，服务于一致性留痕：

1. `object.versioned`
2. `audit.recorded`
3. `review.required`
4. `policy.denied`

### 异步领域事件

提交后由 dispatcher 派发：

1. `asset.created`
2. `asset.deleted`
3. `knowledge.document.created`
4. `knowledge.block.updated`
5. `memory.raw_event.ingested`
6. `memory.candidate.approved`
7. `gateway.call.requested`
8. `backup.completed`

### 读模型刷新事件

1. `index.rebuild.requested`
2. `embedding.refresh.requested`
3. `graph.refresh.requested`
4. `context.invalidate.requested`
5. `notification.dispatch.requested`

## 4. 事务边界

## 4.1 Asset 创建

### 同事务内完成

1. `assets` 插入。
2. `object_registry` 注册。
3. `object_versions` 写入 v1。
4. `audit_events` 写入 `asset_create_requested`。
5. `events` 写入 `asset.created`。
6. `jobs` 中写入一个待执行的 `asset_import_pipeline` job stub。

> **实现状态**: 步骤 1-5 已实现（`mneme/db/assets.py` `create_asset()`）。步骤 6（job stub 写入）尚未实现；当前 asset 创建后不自动触发 pipeline job。

### 事务外完成

1. 文件解析。
2. Markdown 生成。
3. chunk/source map 构建。
4. embedding。
5. graph 更新。

### 失败语义

1. 原件已保存但后处理失败：`asset.status=active`，`knowledge_state=failed`，原件保留。
2. 不回滚原件，不伪造“整个资产不存在”。

## 4.2 Knowledge Block 编辑

### 同事务内完成

1. 写入新的 `object_version`。
2. 更新 `knowledge_blocks` 当前版本指针。
3. `index_states` 标记当前文档为 `stale`。
4. 写入 `block.updated` 事件。
5. 写入审计。

### 事务外完成

1. 重建 chunk。
2. 重建 embedding。
3. 更新 citation。
4. 局部图谱刷新。
5. context pack 失效。

### 读一致性规则

1. 用户永远能读到最新 block 内容。
2. 搜索、graph、citation 在新派生结果就绪前继续使用上一版稳定结果。
3. `index_states` 必须向 UI 明确暴露 `stale_version` 和 `ready_version`。

## 4.3 Memory Candidate 审批

### 同事务内完成

1. `memory_candidates.status` 更新。
2. 创建或更新 `memories`。
3. 写入 `memory_versions`。
4. 写入 `memory_sources`。
5. 写入 `review_items` 决策结果。
6. 写入 `memory.candidate.approved` 事件。
7. 写入审计。

> **实现状态**: 当前 `update_candidate_status()` 仅执行 `candidate_status` 状态迁移（`UPDATE memory_candidates SET candidate_status = ...`），不创建 memories/memory_versions/memory_sources 记录，不写入 audit_events/events。完整的同事务审批链路待实现。

### 事务外完成

1. graph 更新。
2. context cache 失效。
3. notification。

### 硬约束

没有 `activated_by_review_item_id` 的正式 memory 不允许激活；对应 `memory_versions.review_item_id` 必须指向同一次 Review 决策。

## 4.4 Gateway 调用

### 同事务内完成

1. 写入 `api_call_logs` 预留记录，初始状态 `planned`。
2. 写入预算占用记录。
3. 预算占用成功后将 `api_call_logs.status` 推进为 `budget_reserved`。
4. Vault 凭据检查成功后将 `api_call_logs.status` 推进为 `credential_checked`。
5. 写入审计。
6. 如需人工审批，创建 `review_item`；策略拒绝时将 `api_call_logs.status` 推进为 `denied`。

### 事务外完成

1. 真实 provider 调用。
2. fallback。
3. usage 回填。
4. 成功/失败更新。

正式状态机：

```text
planned -> budget_reserved -> credential_checked -> in_flight -> succeeded
planned/budget_reserved/credential_checked -> denied
in_flight -> failed/timeout/cancelled/dead_letter
```

### 关键原则

1. 外部调用前先落审计和预算预留。
2. 外部调用失败后不能让系统“不知道自己调用过”。

## 5. 幂等设计

## 5.1 幂等键类型

| 场景 | 幂等键 |
| --- | --- |
| HTTP 创建资产 | `request_id + sha256 + source` |
| HTTP 提交 memory candidate | `request_id + actor + candidate_hash` |
| Gateway 调用 | `request_id + capability + payload_hash + policy_version` |
| Outbox 消费 | `event_id + consumer_name` |
| Job 执行 | `job_type + aggregate_id + target_version + cause_event_id` |
| Importer 导入 | `import_run_id + source_table + source_pk` |

## 5.2 唯一约束

数据库必须有以下唯一约束：

1. `events(event_id)`
2. `event_deliveries(event_id, consumer_name)`
3. `jobs(job_key)`
4. `api_call_logs(provider_request_fingerprint)` 对支持强幂等的 provider 场景启用（注意：实际列名无 UNIQUE 约束，待修复）
5. `review_items(idempotency_key)` UNIQUE 约束已实现
6. `legacy_mapping_registry(import_run_id, source_table, source_pk)` ✅ 已实现 — migration 0024, DAL 就绪

## 5.3 Worker 重入处理

Worker 重启、超时、重复投递时，处理顺序固定：

1. 先查 `job_key`。
2. 若已有 `succeeded` 结果，直接 ack，不重做。
3. 若已有 `running` 但 lease 未过期，拒绝并稍后重试。
4. 若 lease 过期，抢占并续跑。
5. 若目标版本已被更新版本 supersede，直接标记 `superseded`。

## 6. Queue / Worker / Dead Letter 设计

## 6.1 架构结论

正式采用：

```text
PostgreSQL jobs/events = 事实源
Redis Stream / Redis queue = 派发加速器
Worker = 无状态执行器
DLQ = PostgreSQL dead_letters + Redis dlq topic
```

不采用“任务状态只存在 Redis”。

## 6.2 角色拆分

### Outbox Dispatcher

负责：

1. 轮询 `events.publish_state=pending`。
2. 把 `event_id` 推入 Redis。
3. 写 `event_deliveries` 初始记录。

### Job Scheduler

负责：

1. 从事件生成具体 job。
2. 处理延迟重试。
3. 处理 cron 型任务。

### Worker

负责：

1. 拉取 job_id。
2. 在 PostgreSQL 中 lease job。
3. 心跳续租。
4. 执行并回写结果。

### Retry Sweeper

负责：

1. 扫描 lease 过期 job。
2. 扫描 `retry_at <= now()` 的 job。
3. 把可重试任务重新放入队列。

### Dead Letter Processor

负责：

1. 归档超过最大重试次数的任务。
2. 生成 review item 或系统告警。
3. 提供人工重放入口。

## 6.3 Job 状态机

正式状态词表：

```text
pending / scheduled / running / succeeded / failed / retrying / cancelled / dead_letter / superseded
```

主要迁移：

```text
pending -> scheduled -> running -> succeeded
running -> failed -> retrying -> scheduled
running/failed/retrying -> dead_letter
pending/scheduled/running/retrying -> cancelled
pending/scheduled/running/retrying -> superseded
```

约束：

1. 只有 `pending/scheduled/retrying` 可以被 Worker 领取。
2. Worker 成功领取后必须将状态推进为 `running`，并写入 lease/heartbeat 字段。
3. 只有 `running` 可以写心跳。
4. `succeeded/dead_letter/cancelled/superseded` 为终态。

## 6.4 Dead Letter 分类

| 分类 | 进入 DLQ 条件 | 默认动作 |
| --- | --- | --- |
| `provider_transient_exhausted` | 瞬时错误重试耗尽 | 等待人工重放 |
| `policy_denied_terminal` | 权限/预算/敏感度策略拒绝 | 不重试，进入 Review |
| `payload_invalid` | 上游数据缺失或格式错误 | 回退到修复队列 |
| `code_bug` | 处理器异常或不变量破坏 | P1 告警，禁止自动重放 |
| `external_side_effect_unknown` | 外部调用是否成功无法确定 | 冻结人工确认，避免重复计费 |

## 7. 重试策略

## 7.1 错误分级

### T1 瞬时错误

例：

- Redis 连接抖动
- provider 429 / 503
- 网络超时

策略：

- 指数退避
- 带抖动
- 默认最多 5 次

### T2 可恢复业务错误

例：

- index stale
- graph lock 冲突
- 上游对象暂未 ready

策略：

- 短退避重试
- 最多 8 次
- 目标版本若被 supersede 则终止

### T3 终态错误

例：

- 权限不足
- 敏感级别禁止外发
- review 未通过
- provider 配置缺失

策略：

- 不自动重试
- 直接进入 Review 或失败终态

### T4 不确定副作用错误

例：

- provider 可能已经扣费，但回包丢失

策略：

- 不直接重试原请求
- 先查询 provider 侧 request_id
- 查不到时进入人工确认

## 7.2 默认退避参数

| 任务类型 | 最大重试 | 初始退避 | 上限 |
| --- | --- | --- | --- |
| 索引/图谱/缓存刷新 | 8 | 10s | 30m |
| 外部 provider 调用 | 5 | 30s | 2h |
| 备份/恢复校验 | 3 | 1m | 30m |
| importer delta | 5 | 1m | 1h |

## 8. 回滚与补偿

## 8.1 本地事务回滚

以下情况必须直接数据库回滚：

1. 主对象写失败。
2. audit 写失败。
3. outbox 写失败。
4. review 强依赖对象创建失败。

结论：

`业务写入成功但 audit/outbox 没写进去` 在 Mneme 中视为失败。

## 8.2 异步补偿

### Asset 主线

1. 原件已落地但解析失败：
   - 资产保留。
   - 任务失败。
   - UI 提示“可重试解析”。

### Knowledge 派生链

1. chunk/embedding/citation 失败：
   - 文档正文不回滚。
   - `index_states` 维持 stale。
   - 搜索回退到旧稳定版本或 FTS-only。

### Memory 主线

1. candidate 批准后 graph 刷新失败：
   - memory 仍生效。
   - graph pending repair。

### Gateway 主线

1. provider 成功、回写失败：
   - 记录 `result_received_uncommitted`。
   - 禁止直接重调。
   - 由补偿作业回填结果。

## 8.3 取消语义

取消不是“删除历史”，而是状态迁移：

1. 未执行 job 可 `cancelled`。
2. 已开始的 provider 调用不可假设可撤销，只能标记 `cancel_requested`。
3. 被新版本覆盖的派生任务标记 `superseded`。
4. cancelled/superseded 必须写审计。

## 9. 降级策略

## 9.1 Redis 不可用

处理：

1. `events` 和 `jobs` 仍可落库。
2. dispatcher 转为 DB polling 模式（⚠️ 待实现：当前 `_standby_loop()` 仅 sleep + 重试 Redis lease 获取，不执行 DB polling）。
3. 低成本本地任务可同步或半同步执行。
4. 外部高成本任务暂停出队。

用户感知：

- 写入可成功，但派生更新延迟。

## 9.2 Worker 不可用

处理：

1. 允许资产入库、知识编辑成功落库。
2. UI 明确显示 `pending_processing`。
3. 所有长链路保持可重放。

## 9.3 Gateway 不可用

处理：

1. 优先 fallback 到本地 provider。
2. 没有本地 provider 时，把任务送入 Review 或 `awaiting_provider`。
3. 禁止业务线程直接绕过 Gateway。

## 9.4 Graph 不可用

处理：

1. Graph 刷新失败不阻断 Asset/Knowledge/Memory 主链。
2. Graph UI 显示“图谱滞后于版本 X”。

## 9.5 Context Compiler 不可用

处理：

1. Agent 查询回退为 `knowledge search + memory search`。
2. 不阻断基础搜索。

## 9.6 Review 不可用

处理：

1. 高敏感、高成本、高风险操作一律冻结。
2. 普通低风险读操作继续。
3. 需要审批的对象停留在 `pending_review`。

## 10. 跨模块一致性保证

## 10.1 统一关联链

所有跨模块链路必须带：

1. `request_id`
2. `correlation_id`
3. `causation_id`
4. `actor_id`
5. `target_version`

这样可把：

```text
HTTP 请求
→ 业务对象
→ 审计
→ 事件
→ job
→ provider call
→ review decision
→ context invalidation
```

完整串起来。

## 10.2 版本钉住规则

任何派生结果都必须钉住上游版本：

| 派生对象 | 必须记录 |
| --- | --- |
| chunk | `document_version` |
| embedding | `chunk_version` 或 `document_version` |
| graph edge | `source_object_version` |
| context pack | `knowledge_version_set + memory_version_set` |
| review item | `target_version` |

禁止“只知道对象 ID，不知道依据哪个版本计算出来”。

## 10.3 主归属规则

1. Inbox 不是长期主对象库，正式对象进入 Asset/Raw Event 后必须有主归属。
2. Asset 是原件与文件元数据主归属。
3. Knowledge 是文档结构化与可引用内容主归属。
4. Memory 是经 Review 后的长期事实主归属。
5. Review 是横切治理中台，不归属于 Memory 专属。
6. Gateway 只解析 capability，不拥有业务对象主归属。

## 11. 审计与一致性联动

以下动作没有审计即视为未完成：

1. 登录和 token 使用。
2. 敏感数据访问。
3. provider 调用。
4. review 决策。
5. backup/restore。
6. importer dry-run 和正式导入。
7. rollback / replay / dead-letter 重放。

审计写入失败时，业务事务必须失败；日志系统失败不等同于审计失败。

## 12. 验收标准

开工前，相关实现必须满足：

1. 每个写 API 都有明确事务边界说明。
2. 每个异步 consumer 都定义幂等键。
3. `events` 与 `audit_events` 与主业务对象同事务提交。
4. Redis 故障演练下，系统仍能落库并恢复派发。
5. Worker 重启演练下，不出现重复正式 memory、重复 provider 计费、重复 graph 写入。
6. `block.updated` 演练下，旧搜索结果可读、新版本最终可追平。
7. 所有 DLQ 项都能在 UI 看到失败原因、重试建议和相关对象。

## 13. 最终结论

Mneme v4.1.1 的一致性设计不靠”所有东西同步完成”，而靠：

```text
事务内写主事实
+ 同事务写审计与 outbox
+ PostgreSQL 持有 job 真相
+ Redis 只做加速
+ 版本钉住派生结果
+ 用重试、补偿、降级把不确定外部调用关进笼子
```

只要这几条被严格执行，Asset、Knowledge、Memory、Review、Gateway、Context、Graph 就不会在开发中各写一套控制流，更不会在上线后变成黑箱。
